# Code Review

Date: 2026-05-09

Scope: review of the current codebase after follow-up fixes for refresh thread handling, config backup/repair, and diagnostics.

Build status: checked with Visual Studio Developer environment and the existing CMake/Ninja build directory. Result: `ninja: no work to do`.

Status values: `Resolved`, `Partially resolved`, `Open`.

## Findings

### 1. Medium: `OnRefreshDone()` still performs a UI-thread `join()`

Status: `Resolved`

`TrayApp::OnRefreshDone()` joins the refresh worker after the refresh message is received. In the current implementation the worker normally posts the message as its final action, so the join should be quick. The remaining risk is that future worker-side cleanup or timing changes can make the UI thread wait in the message handler.

Recommendation:

- Avoid blocking joins in the UI message handler.
- Use a persistent worker, an event-based completion model, or another ownership model where completed workers can be cleaned up without waiting on network work from the UI thread.

### 2. Medium: copy-only config backup still allows overwriting the original

Status: `Resolved`

`BackupConfig()` treats a successful `copy_file()` fallback as a successful backup. If `rename()` fails but `copy_file()` succeeds, the original `config.json` remains in place and can later be overwritten by `Save()`. This is not direct data loss because a backup exists, but the behavior should be explicit.

Recommendation:

- Make backup result explicit: `Moved`, `Copied`, or `Failed`.
- If the result is `Copied`, either document that a verified copy is enough before overwrite, or avoid overwriting the original unless it was moved.
- Optionally verify backup file size/hash before saving repaired config.

### 3. Medium: tray icon restore after Explorer restart is minimal

Status: `Resolved`

On `TaskbarCreated`, the app calls `AddTrayIcon()`, which restores the initial offline/launching icon and tooltip instead of the latest known tray state. The code also does not call `NIM_SETVERSION`. After Explorer restarts, the tray can show stale state until the next refresh.

Recommendation:

- Store the last tray icon/tooltip state and restore it when re-adding the icon.
- Or trigger an immediate refresh after re-adding the tray icon.
- Consider setting `NOTIFYICON_VERSION_4` with `NIM_SETVERSION`.

### 4. Low: tray/window setup ignores several WinAPI return values

Status: `Resolved`

Several setup calls are not checked, including `RegisterClassExW`, `CreateWindowExW`, `Shell_NotifyIconW`, and `SetTimer`. If any of these fail, the app can continue in a degraded state with little diagnostic information.

Recommendation:

- Check key WinAPI return values during startup and tray setup.
- Add `OutputDebugStringW` diagnostics for failures.
- Retry tray icon registration when appropriate.

### 5. Low: `HttpGet` ignores timeout option failures

Status: `Resolved`

`WinHttpSetOption` calls for resolve/connect/send/receive timeouts are not checked. If any timeout option fails to apply, network requests may not observe the expected timeout behavior.

Recommendation:

- Check the return value of each `WinHttpSetOption`.
- Throw or log a diagnostic if timeout configuration fails.

### 6. Low: IP provider fallback errors are silent

Status: `Resolved`

`IpProvider` suppresses all provider-level errors while trying fallback providers. This is good for user experience, but weak for troubleshooting provider outages or unexpected response formats.

Recommendation:

- Log provider failures with `OutputDebugStringA/W`.
- Keep fallback behavior unchanged for normal users.

## Suggested Fix Order

1. Remove or redesign the remaining UI-thread join in `OnRefreshDone()`.
2. Make config backup results explicit and align behavior/documentation.
3. Restore current tray state after Explorer restart.
4. Add startup/tray WinAPI diagnostics.
5. Check WinHTTP timeout option calls.
6. Add debug diagnostics for IP provider fallback failures.

# Ревью кода

Дата: 2026-05-09

Область проверки: текущее состояние проекта после исправлений refresh thread handling, backup/repair конфигурации и диагностики.

Статус сборки: проверено через Visual Studio Developer environment и существующую директорию CMake/Ninja `build`. Результат: `ninja: no work to do`.

Значения статусов: `Resolved`, `Partially resolved`, `Open`.

## Найденные проблемы

### 1. Medium: `OnRefreshDone()` всё ещё выполняет `join()` в UI-потоке

Статус: `Resolved`

`TrayApp::OnRefreshDone()` присоединяет refresh-worker после получения сообщения о завершении обновления. В текущей реализации worker обычно отправляет сообщение последним действием, поэтому `join()` должен быть быстрым. Оставшийся риск в том, что будущая cleanup-логика worker-а или изменения тайминга могут заставить UI-поток ждать внутри обработчика сообщения.

Рекомендации:

- Избегать блокирующих `join()` в UI message handler.
- Использовать постоянный worker, event-based completion model или другую модель владения, где завершённые worker-ы можно очищать без ожидания сетевой работы из UI-потока.

### 2. Medium: copy-only backup конфигурации всё ещё допускает перезапись оригинала

Статус: `Resolved`

`BackupConfig()` считает успешный fallback через `copy_file()` успешным backup. Если `rename()` не удался, но `copy_file()` прошёл успешно, исходный `config.json` остаётся на месте и позже может быть перезаписан через `Save()`. Это не прямая потеря данных, потому что backup существует, но поведение лучше сделать явным.

Рекомендации:

- Сделать результат backup явным: `Moved`, `Copied` или `Failed`.
- Если результат `Copied`, либо задокументировать, что проверенной копии достаточно перед overwrite, либо не перезаписывать оригинал, если он не был перемещён.
- Опционально проверять размер/hash backup-файла перед сохранением исправленного config.

### 3. Medium: восстановление tray-иконки после перезапуска Explorer минимальное

Статус: `Resolved`

На `TaskbarCreated` приложение вызывает `AddTrayIcon()`, который восстанавливает начальную offline/launching иконку и tooltip вместо последнего известного состояния трея. Также не вызывается `NIM_SETVERSION`. После перезапуска Explorer трей может показывать устаревшее состояние до следующего refresh.

Рекомендации:

- Сохранять последнее состояние tray icon/tooltip и восстанавливать его при повторном добавлении иконки.
- Или запускать немедленный refresh после повторного добавления иконки.
- Рассмотреть установку `NOTIFYICON_VERSION_4` через `NIM_SETVERSION`.

### 4. Low: setup tray/window игнорирует часть return values WinAPI

Статус: `Resolved`

Некоторые setup-вызовы не проверяются: `RegisterClassExW`, `CreateWindowExW`, `Shell_NotifyIconW`, `SetTimer`. Если один из них завершится ошибкой, приложение может продолжить работу в деградированном состоянии почти без диагностики.

Рекомендации:

- Проверять ключевые return values WinAPI на старте и при настройке трея.
- Добавить диагностику через `OutputDebugStringW`.
- Повторять регистрацию tray icon там, где это уместно.

### 5. Low: `HttpGet` игнорирует ошибки установки timeout options

Статус: `Resolved`

Вызовы `WinHttpSetOption` для resolve/connect/send/receive timeout не проверяются. Если какой-то timeout option не применился, сетевые запросы могут вести себя не так, как ожидается.

Рекомендации:

- Проверять return value каждого `WinHttpSetOption`.
- Бросать исключение или писать diagnostic log, если настройка timeout не удалась.

### 6. Low: ошибки fallback IP-провайдеров остаются тихими

Статус: `Resolved`

`IpProvider` подавляет все provider-level ошибки при переборе fallback-провайдеров. Это хорошо для пользовательского опыта, но слабо для диагностики недоступности провайдера или неожиданного формата ответа.

Рекомендации:

- Логировать ошибки провайдеров через `OutputDebugStringA/W`.
- Оставить fallback-поведение без изменений для обычных пользователей.

## Рекомендуемый порядок исправлений

1. Убрать или переработать оставшийся UI-thread `join()` в `OnRefreshDone()`.
2. Сделать результат backup конфигурации явным и согласовать поведение с документацией.
3. Восстанавливать текущее состояние tray после перезапуска Explorer.
4. Добавить диагностику WinAPI startup/tray ошибок.
5. Проверять вызовы WinHTTP timeout options.
6. Добавить debug-диагностику ошибок fallback IP-провайдеров.
