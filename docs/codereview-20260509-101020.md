# Code Review

Date: 2026-05-09

Scope: review of the current codebase after portable autorun handling, single-instance protection, and legacy tray tooltip/menu behavior fixes.

Build status: checked with the existing Visual Studio/CMake/Ninja build directory before the fixes, then rebuilt after the fixes. Result: build completed successfully.

Status values: `Resolved`, `Partially resolved`, `Open`.

## Findings

### 1. Medium: application exit can block while a refresh request is running

Status: `Open`

`TrayApp::~TrayApp()` sets the shutdown flag and then joins the refresh worker thread. If the worker is currently inside a WinHTTP request, the UI can wait until the HTTP timeout expires. This is not data-corrupting, but it can make the app feel frozen during exit.

Recommendation:

- Add cancellable HTTP requests or move refresh work to an ownership model that can be shut down without blocking the UI.
- Keep the current join until a safe cancellation path exists, because detaching the worker would risk use-after-free access to `TrayApp` members.

### 2. Medium/Low: autorun enabling assumes the Run registry key already exists

Status: `Resolved`

`AppSettings::SetAutorun()` used `RegOpenKeyExW()` with `KEY_SET_VALUE`. The Run key usually exists, but using `RegCreateKeyExW()` is more robust for portable usage and unusual user profiles.

Recommendation:

- Use `RegCreateKeyExW()` when enabling or disabling autorun.

### 3. Low: startup continued after the tray host window failed to initialize

Status: `Resolved`

If the hidden tray host window could not be created, the app continued with tray icon setup, refresh startup, and timer creation. This could leave a broken invisible process.

Recommendation:

- Fail startup explicitly when the tray host window cannot be created.
- Show a user-visible error instead of leaving the process running in a degraded state.

### 4. Low: autorun path parsing had fixed-size buffers and weak unquoted command handling

Status: `Resolved`

Executable path detection used `MAX_PATH`, registry command reading used a fixed 1024-character buffer, and unquoted autorun commands with spaces could be parsed incorrectly.

Recommendation:

- Use dynamically sized buffers for executable path, full path normalization, registry value reading, and environment expansion.
- Prefer quoted autorun commands, but parse existing unquoted `.exe` commands more defensively.

### 5. Low: tray popup menu omitted the standard post-menu `WM_NULL`

Status: `Resolved`

The tray menu used `SetForegroundWindow()` before `TrackPopupMenu()`, but did not post `WM_NULL` afterwards. Posting `WM_NULL` is a common Win32 tray-menu pattern that reduces the chance of sticky popup behavior.

Recommendation:

- Call `PostMessageW(hWnd, WM_NULL, 0, 0)` after `TrackPopupMenu()`.

# Ревью кода

Дата: 2026-05-09

Область проверки: текущее состояние проекта после исправлений portable-автозапуска, защиты от повторного запуска и возврата legacy-поведения tooltip/menu в трее.

Статус сборки: проверено через существующую директорию сборки Visual Studio/CMake/Ninja до исправлений, затем пересобрано после исправлений. Результат: сборка завершилась успешно.

Значения статуса: `Resolved`, `Partially resolved`, `Open`.

## Найденные проблемы

### 1. Medium: выход из приложения может блокироваться, пока выполняется refresh-запрос

Статус: `Open`

`TrayApp::~TrayApp()` выставляет флаг завершения и затем ждёт завершения refresh-потока через `join()`. Если поток в этот момент находится внутри WinHTTP-запроса, UI может ждать до истечения HTTP timeout. Это не приводит к повреждению данных, но может выглядеть как зависание при выходе.

Рекомендация:

- Добавить отменяемые HTTP-запросы или изменить модель владения refresh-работой так, чтобы завершение не блокировало UI.
- До появления безопасной отмены сохранить текущий `join`, потому что отсоединение потока может привести к обращению к уже уничтоженным полям `TrayApp`.

### 2. Medium/Low: включение автозапуска предполагало, что registry key Run уже существует

Статус: `Resolved`

`AppSettings::SetAutorun()` использовал `RegOpenKeyExW()` с `KEY_SET_VALUE`. Обычно ключ Run существует, но для portable-приложения и нетипичных пользовательских профилей надёжнее использовать `RegCreateKeyExW()`.

Рекомендация:

- Использовать `RegCreateKeyExW()` при включении и выключении автозапуска.

### 3. Low: запуск продолжался после ошибки инициализации hidden window для tray

Статус: `Resolved`

Если hidden window для tray не создавался, приложение продолжало регистрацию tray icon, запуск refresh и создание таймера. Это могло оставить сломанный невидимый процесс.

Рекомендация:

- Явно прерывать запуск, если tray host window не удалось создать.
- Показывать пользователю понятную ошибку вместо продолжения в деградированном состоянии.

### 4. Low: разбор пути автозапуска использовал фиксированные буферы и слабо обрабатывал некавыченые команды

Статус: `Resolved`

Определение пути exe использовало `MAX_PATH`, чтение registry-команды использовало фиксированный буфер на 1024 символа, а некавыченые autorun-команды с пробелами могли разбираться неправильно.

Рекомендация:

- Использовать динамические буферы для пути exe, нормализации полного пути, чтения registry value и раскрытия environment-переменных.
- Продолжать записывать autorun-команду в кавычках, но более устойчиво читать существующие некавыченые `.exe`-команды.

### 5. Low: после popup-menu в трее не отправлялся стандартный `WM_NULL`

Статус: `Resolved`

Меню tray вызывало `SetForegroundWindow()` перед `TrackPopupMenu()`, но не отправляло `WM_NULL` после закрытия меню. Это распространённый Win32-паттерн, который снижает риск “залипания” popup-menu.

Рекомендация:

- Вызывать `PostMessageW(hWnd, WM_NULL, 0, 0)` после `TrackPopupMenu()`.
