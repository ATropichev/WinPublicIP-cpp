# Code Review

Date: 2026-05-09

Scope: review of the current codebase after the latest review fixes and commit series.

Build status: checked with the existing Visual Studio/CMake/Ninja build directory. Result: `ninja: no work to do`.

Status values: `Resolved`, `Partially resolved`, `Open`.

## Findings

### 1. Medium: the app could keep running without a tray icon

Status: `Resolved`

`TrayApp::AddTrayIcon()` logged `Shell_NotifyIconW(NIM_ADD)` failures and returned, while the constructor still started refresh work and the timer. Because this is a tray-only application, running without a tray icon leaves the user without the normal menu and exit path.

Recommendation:

- Make tray icon registration a startup requirement.
- Stop startup with a user-visible error if the tray icon cannot be added.

### 2. Medium/Low: startup exceptions after tray icon registration could leak the tray icon

Status: `Resolved`

If an exception occurred after the tray icon was successfully added but before the `TrayApp` constructor completed, the destructor would not run and the tray icon could remain registered.

Recommendation:

- Track whether the tray icon was registered.
- Clean up the icon and window before rethrowing constructor-time startup failures.

### 3. Medium/Low: exit can still block while a refresh request is running

Status: `Open`

`TrayApp::~TrayApp()` still joins the refresh worker. This avoids use-after-free risks, but can make exit wait until an in-flight WinHTTP request finishes or times out.

Recommendation:

- Address this separately with cancellable HTTP requests or a worker model that does not require blocking UI shutdown on network work.

### 4. Low: unquoted autorun command parsing was too broad

Status: `Resolved`

The fallback parser accepted the first `.exe` substring in an unquoted autorun command. This could misidentify rare paths where `.exe` appears in a directory name before the actual executable filename.

Recommendation:

- Keep writing quoted autorun commands.
- For existing unquoted values, only accept an `.exe` candidate if it exists as a regular file.

# Ревью кода

Дата: 2026-05-09

Область проверки: текущее состояние проекта после последних исправлений и серии коммитов.

Статус сборки: проверено через существующую директорию сборки Visual Studio/CMake/Ninja. Результат: `ninja: no work to do`.

Значения статуса: `Resolved`, `Partially resolved`, `Open`.

## Найденные проблемы

### 1. Medium: приложение могло продолжить работу без иконки в трее

Статус: `Resolved`

`TrayApp::AddTrayIcon()` логировал ошибку `Shell_NotifyIconW(NIM_ADD)` и возвращался, но конструктор всё равно запускал refresh и timer. Для tray-only приложения это плохой режим: пользователь остаётся без обычного меню и штатного выхода.

Рекомендация:

- Считать регистрацию tray icon обязательной частью запуска.
- Прерывать запуск с понятным сообщением пользователю, если иконку не удалось добавить.

### 2. Medium/Low: исключения после регистрации tray icon могли оставить иконку без cleanup

Статус: `Resolved`

Если исключение происходило после успешного добавления tray icon, но до завершения конструктора `TrayApp`, деструктор не вызывался бы и иконка могла остаться зарегистрированной.

Рекомендация:

- Отслеживать факт успешной регистрации tray icon.
- При ошибках запуска внутри конструктора удалять иконку и окно перед повторным выбросом исключения.

### 3. Medium/Low: выход всё ещё может блокироваться, пока выполняется refresh-запрос

Статус: `Open`

`TrayApp::~TrayApp()` всё ещё ждёт refresh-поток через `join()`. Это защищает от use-after-free, но может заставить выход ждать, пока WinHTTP-запрос завершится или истечёт timeout.

Рекомендация:

- Исправлять отдельно через отменяемые HTTP-запросы или worker-модель, где UI shutdown не блокируется на сетевой операции.

### 4. Low: разбор некавыченой autorun-команды был слишком широким

Статус: `Resolved`

Fallback-парсер принимал первую подстроку `.exe` в некавыченой autorun-команде. Это могло ошибиться в редких путях, где `.exe` встречается в имени директории до настоящего исполняемого файла.

Рекомендация:

- Продолжать записывать autorun-команды в кавычках.
- Для существующих некавыченых значений принимать `.exe`-кандидат только если он существует как regular file.
