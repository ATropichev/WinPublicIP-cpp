# Code Review

Date: 2026-05-09

Scope: review of the current codebase after tray startup and autorun parsing hardening.

Build status: checked with the existing Visual Studio/CMake/Ninja build directory. Result: `ninja: no work to do`.

Status values: `Resolved`, `Partially resolved`, `Open`.

## Findings

### 1. Medium: Explorer restart restore ignored tray icon registration failure

Status: `Resolved`

The `TaskbarCreated` handler called `AddTrayIcon()` but ignored its return value. If Explorer restarted and the icon could not be re-added, the process would keep running and refresh work would continue without visible tray UI.

Recommendation:

- Check the restore result.
- Do not start an immediate refresh when the tray icon cannot be restored.
- Log the restore-specific failure for diagnostics.

### 2. Medium/Low: `trayIconAdded_` could stay true after Explorer lost the icon

Status: `Resolved`

After Explorer restarts, the old tray icon registration is no longer valid, but `trayIconAdded_` still represented the pre-restart state. If re-registration failed, the in-memory flag could say the icon exists while the shell no longer shows it.

Recommendation:

- Treat `TaskbarCreated` as a lost shell registration event.
- Reset the in-memory tray icon state before attempting to add the icon again.

### 3. Medium/Low: exit can still block while a refresh request is running

Status: `Open`

`TrayApp::~TrayApp()` still joins the refresh worker. This protects object lifetime, but can make shutdown wait for an in-flight WinHTTP request.

Recommendation:

- Handle separately with cancellable HTTP requests or a worker model that does not block UI shutdown on network operations.

### 4. Low: primary and restore tray icon flows shared one method with unclear semantics

Status: `Resolved`

`AddTrayIcon()` was used both for first startup and Explorer restart recovery. The two flows need different behavior: startup failures should abort the app, while restore failures should be logged and retried later through future shell events/user actions.

Recommendation:

- Keep `AddTrayIcon()` as the low-level registration function.
- Add a dedicated restore path for `TaskbarCreated`.

# Ревью кода

Статус сборки: проверено через существующую директорию сборки Visual Studio/CMake/Ninja. Результат: `ninja: no work to do`.

Значения статуса: `Resolved`, `Partially resolved`, `Open`.

## Найденные проблемы

### 1. Medium: восстановление после рестарта Explorer игнорировало ошибку регистрации tray icon

Статус: `Resolved`

Обработчик `TaskbarCreated` вызывал `AddTrayIcon()`, но игнорировал результат. Если Explorer перезапускался и иконку не удавалось добавить заново, процесс продолжал работать и запускал refresh без видимого tray UI.

Рекомендация:

- Проверять результат восстановления.
- Не запускать немедленный refresh, если tray icon не восстановлен.
- Логировать ошибку именно сценария восстановления.

### 2. Medium/Low: `trayIconAdded_` мог остаться true после потери иконки Explorer

Статус: `Resolved`

После рестарта Explorer старая регистрация tray icon уже недействительна, но `trayIconAdded_` всё ещё описывал состояние до рестарта. Если повторная регистрация не удалась, внутренний флаг мог показывать, что иконка есть, хотя shell её уже не отображает.

Рекомендация:

- Считать `TaskbarCreated` событием потери shell-регистрации.
- Сбрасывать внутреннее состояние tray icon перед повторным добавлением.

### 3. Medium/Low: выход всё ещё может блокироваться, пока выполняется refresh-запрос

Статус: `Open`

`TrayApp::~TrayApp()` всё ещё ждёт refresh-поток через `join()`. Это защищает время жизни объектов, но может заставить shutdown ждать активный WinHTTP-запрос.

Рекомендация:

- Исправлять отдельно через отменяемые HTTP-запросы или worker-модель, где UI shutdown не блокируется на сетевой операции.

### 4. Low: первичное добавление и восстановление tray icon использовали один метод с неочевидной семантикой

Статус: `Resolved`

`AddTrayIcon()` использовался и при первом запуске, и при восстановлении после рестарта Explorer. Поведение этих сценариев отличается: ошибка при старте должна прерывать приложение, а ошибка восстановления должна логироваться и ждать следующей попытки.

Рекомендация:

- Оставить `AddTrayIcon()` низкоуровневой регистрацией.
- Добавить отдельный путь восстановления для `TaskbarCreated`.
